FAQ

What is Instance?

Instance is a browser UEFI — a firmware layer that boots beneath your app and resolves the messy abstractions of traditional frontend development: the build-tool ceremony, the split between the class, the DOM node, and reactive state. In layman's terms, a fusion runtime. It is built for the backend systems engineer — someone fluent in predictable tools who has watched rendering a box on a screen turn into a five-hundred-package ritual, and wants none of it.

Ironically, such aversion is not merely aesthetic or cultural—it's fundamentally correct. The Shai-Hulud worm campaigns (2025–2026) have proven that build tools and dependency managers are not just overhead—they are a massive cybersecurity loophole. Attackers threaded the needle by poisoning the npm ecosystem itself, compromising over 700 packages through a self-replicating worm that stole credentials, republished malicious packages, and spread across maintainers and repositories. Each npm install became a potential entry point. Each transitive dependency became a blind trust. The key takeaway: Build tools are only as 'safe' as your dependency chain is.

The core architectural solution: making build tools optional with Direct Element Architecture. The element is the framework (instance). new Div() returns a real HTMLDivElement, minted through the browser platform itself, that is also an instance of your class. No dependencies, no proxy wrapper, no virtual node, no handle (i.e Component.el), no W3C Custom Elements, no build step. Nothing sits between you and the element. 

let $div = new Div('@#target', 'Hello world!'); // NEW DIV AT TARGET, SET TEXT 'Hello world!'
console.log($div); // <div>

Is it complicated?

Low skill floor, high skill ceiling. The floor is the line above. The ceiling is everything else in this reference — a reactive kernel, a live-DOM registry, an event system, engine mode, the well-known-symbol protocol. Instance has been called dense, and the depth is real. But the depth is optional: you reach for it when a problem needs it, and never before. Most webpages never need to enter the 'deep end'.

Do I need a build step?

No. Instance ships as one classic <script> — no bundler, no dependency graph, no node server to render a div. Tooling exists for people who want it (a <script type="instance"> / .is dialect, a Vite plugin) and is covered under Tooling & Environment, but it is never required.

Does it use Web Components / Custom Elements?

No. Instance reaches the "class instance is the element" result without the W3C Custom Elements registry — the mechanism it was widely assumed to require. The requirement for custom element hyphenation (aesthetics) as well as Safari's lack of [is] support (vendor fragmentation) on programmatic extensible native elements made bypassing Custom Elements mandatory. See Identity for how.

How does it relate to jQuery?

Instance is the spiritual successor to jQuery, written for the problems of 2026. It even merges jQuery.fn where present — but where the two disagree on a name, Instance keeps its meaning. See Interop.

More questions — why a classic script, where the virtual DOM went, SSR, TypeScript — are answered under FAQ — deeper cuts.

Dependencies: 0

One file, no pipeline

Instance ships as a single classic <script>. There is no (external) compiler you must run, no bundler, no package graph. The script boots off its own tag attributes (document.currentScript) and is ready synchronously; InstanceReady resolves for code that wants a promise anyway. Tooling (a Vite plugin) exists for people who want it — it is never required.

The element is the instance

Class syntax is the API. A class describes an element; construction mints the element. Methods are element methods. Fields prefixed $, $$ are reactive state stored beside the element (weakly, keyed on it), never on it — the DOM node stays clean.

Live-DOM registry

A class knows its living instances — Row.all(), Row[0] — but the registry tracks the document, not your allocations. Detach an element and it leaves the registry (unqueryable, though the element itself stays alive in your hands); reattach it and it returns, remembering its slot. Dispose it and it is forgotten forever. Nothing is pinned: disposal reclaims ~98% in profiling, flat across create/destroy cycles.

Events are the pulse face of signals

One propagation kernel carries both faces. Behaviors ($fields) have a value at rest, equality-dedup, and replay-on-subscribe. Pulses (DOM events) have none of those: every occurrence lands, late subscribers hear nothing old, and handlers run atomically — re-entrant fires queue and deliver in order. Same kernel, two faces, each with the semantics its face requires.

House wins

When Instance and a foreign library disagree about a name, Instance keeps its meaning. The jQuery merge grafts every jQuery.fn verb it can — but on, remove, find and friends stay Instance's. Interop is a guest, not a landlord.

Structural honesty

Where Instance diverges from the platform it says so, and where it can close the gap it measures the price first. instanceof Instance is true because the mint is the brand. In engine mode even Object.getPrototypeOf(el) answers natively — for a measured ~1.16× tax on hot reads that you opt into, never pay by default.

Tooling & Environment

Instance needs nothing but a script tag — but it meets you where you work. This section covers booting the runtime, the naming policy for the tag vocabulary, and the optional authoring paths (<script type="instance">, the .is dialect, the Vite plugin) that projects are expected to reach for as they grow.

<script src="./Instance-0.87.z.js" ⚡ autoclass> boot

Drop the file in with a classic script tag. Boot options are attributes on the tag itself — Instance reads document.currentScript:

  <script src="./Instance-0.87.z.js" ⚡ autoclass="core, baseline"></script>

Attribute sigils. Boot attributes may be written with an @ sigil form (@pascalite, @uppercase) as shorthand for typeset="…"; the whitelist is the same vocabulary, deduped and normalized.

After the script executes, the tag vocabulary is global (Div, Span, Section, THead, …), the event vocabulary is global (Click, MouseDown, KeyUp, …), and window.Instance is the framework class. InstanceReady is a promise resolving when boot completes — module code and late loaders can await it.

await InstanceReady;
  document.body.appendChild(new Div());
Module code? Retrieve Instance without the named global through the IVC broker: const Instance = globalThis[Symbol.for('Instance')](); The Vite plugin wraps this as import Instance, { ready } from 'virtual:instance'.

Authoring paths — <script type="instance">, .is, Vite

Plain scripts are the floor; for 'real' projects the AOT compiler earns its place. Instance recognizes Instance-typed scripts and .is files and compiles them at boot — the reactive sigils, lifecycle blocks, and the .is dialect become available without a separate build step. These are the expected way to write anything beyond a snippet.

<!-- compiled inline: sigils, @mount, effect() all work -->
  <script type="instance">
    class Counter extends Div {
      $count = 0
      render$() { this.textContent = `count: ${this.$count}` }
    }
  </script>

  <!-- or an external .is file, compiled on load -->
  <script src="/app.is"></script>

Construction

new Div( [init] ) ⇒ HTMLDivElement

Description: Mint a real element of the class's tag. The returned value is the DOM node itself — attach it, style it, query it like any element, because it is one.

const box = new Div();
  box.className = 'card';
  document.body.appendChild(box);

  box instanceof Div;             // true
  box instanceof HTMLDivElement;  // true
  box.tagName;                    // "DIV"

Every HTML tag has a Pascalite-named class: Div, Span, Table, Section, Article — and the acronym tags resolve the way the house table says, not the way naive casing would: tr → TR, thead → THead, tbody → TBody, td → TD.

new Div( '@expression' ) ⇒ Element (placed) — the DSL

Description: The first string argument is a positional expression in a small natural-language DSL: sigil, optional verb, optional ordinal phrase, container. Construction and placement become one move.

Sigils

SigilMeaning
@singular — the expression names one place
@@plural — the container phrase matches all; a single constructed element takes the first match
%Name / @%Namereserved — the instance selector (select by Instance class). Parses today; resolution throws its own honest error: [DSL] Instance selector (%) resolution is not yet implemented

Verbs

Six placement verbs, each taking a selector (with optional connector sugar — to / within / of — between verb and selector):

VerbPlacement
@selectorlast child of the target
@append [to] selectorlast child of the target
@prepend selectorfirst child of the target
@before selectorprevious sibling of the target
@after selectornext sibling of the target
@replace selectorreplaces the target
@wrap selectorwraps the target — the new element takes the target's position and adopts it as a child
new Div('@append main');
  new Aside('@prepend to #layout');
  new Div('@wrap #logo');           // logo now lives inside the new div, in place

Verbless: the ordinal slot

With no verb, the expression is the placement — the new element takes the named slot. The grammar: [the] [every] <ordinal> [from last] [depth] [filter] of <container>. Ordinals run first … tenth plus last; one-word firstchild/lastchild normalize.

new Div('@third child of #wrap');   // becomes the 3rd child — measured: index 2
  new Div('@last child of #wrap');    // appends
  new Div('@the first child of nav'); // 'the' is optional garnish

Depth words

child, grandchild, great-grandchild, great-great-… — each great- descends one more generation. The slot is taken at that depth under the container:

new Div('@first grandchild of #main');  // first child of #main's first child

Filters

A selector between the ordinal phrase and of scopes the counting to matching children only:

new Div('@third .kid of #wrap');    // the slot of the third child MATCHING .kid —
                                      // non-matching siblings don't count

Ancestor arrows

Inside a container phrase, arrows constrain the match by ancestry: sel < ancestor (immediate parent), sel << ancestor (any depth). The route arrows <= / <<= parse and are reserved for the routing layer.

new Div('@.kid < #wrap');   // append into the .kid whose PARENT is #wrap

Grammar the parser also accepts

every/each mark plurality (@each .row); every <ordinal> is ordinal repetition grammar (@every third child of … — with a single @ element, the first such slot is taken). from last counts the ordinal from the tail (@second from last child of …).

Selectors only, loud errors. Targets are selector strings — an element reference throws a [DSL] error naming the rule, as does an unknown ordinal, a dangling of, or an empty phrase. The parser's errors are the grammar's documentation; trust them.
Content rides along. 0.87.z A plain string beside a DSL expression becomes the element's text: new Div('@append main', 'hello world') constructs, places, and fills. Order-agnostic. Without a route, a plain string keeps its long-standing meaning — the element's name.

await new Div( ) ⇒ Element (settled)

Description: Elements are awaitable. Construction with pending asynchronous work (async members settling, queued sequenced work) exposes a virtual then; awaiting resolves to the element itself once its settle chain drains. An element with nothing pending awaits straight through — await new Div() is always safe and always hands back the element.

const el = await new Div();          // no pending work — immediate, el is the div

  class Loader extends Div {
    async $data() { return fetch('/api').then(r => r.json()); }  // async member
  }
  const l = await new Loader();        // resolves when $data settles

Boot directly into it with the ⚡ script attribute. The splice is per-tag — powering a table-header class can never leak into tbody, because every tag has its own prototype link (see above).

The price, measured. Engine mode inserts one link into hot native prototype chains: ~1.16× on warm DOM property reads in Chrome (a per-element proxy design measures 1.78×; the create/update paths benchmarks actually race are untouched). It also assumes page ownership — every element of a powered tag on the page, including ones you didn't create, inherits through the spliced chain. Engine mode is for apps that own their document; it is never on by default.

Well-Known Symbols

Instance implements a slice of JavaScript's well-known symbol protocol directly on elements and element classes — the same hooks (Symbol.iterator, Symbol.hasInstance, …) that make native objects feel native. They are spliced onto the framework through one relay, so every element and every tag-class answers the protocols below. A few are intentionally left as stubs (present, but empty) — those are marked.

Two receivers. Most of these dispatch on typeof this: an element (object) answers about itself; a class (function) answers about its live instance collection. That split is the recurring shape of the whole protocol surface.
Well-known Symbols are an extension of the Symbol protocol. They are an abstract inbuilt JavaScript metaprogramming interface that enables the fine grained customization of, and custom overrides to, how any object or function assigned such Symbols should interact with its environment under certain 'template' lexical conditions. They are scope-aware, highest-precedence behavioral interface overrides, which TC39 designed specifically to enable the ability for programmers to override the default behavior of object-ish variables without breaking backwards compatibility with the existing web. Instance just actually uses them.

Symbol.iterator — iterate the collection

Implemented. Elements and, more usefully, Collections are iterable — Class.all(), Class.where(fn), and the results of the string protocol all yield their elements to for…of and spread.

for (const card of Card.all()) card.classList.add('seen');
  const els = [...Card.all()];          // an array of every live Card
  const hits = [...'apple'.match(Card)]; // spread a match Collection
Spreading the bare class ([...Card]) is not yet supported — go through Card.all(), which returns the iterable Collection.

Symbol.hasInstance — the brand behind instanceof

Implemented. el instanceof Instance is true because the mint is the brand (not a prototype-chain link). Tag classes keep ordinary instanceof so x instanceof Div stays tag-specific. See Identity for the full story.

box instanceof Div;       // true (tag-specific)
  box instanceof Instance;  // true (the brand)

Symbol.toPrimitive — coercion, dispatched by receiver

Implemented. Coercion is meaningful and receiver-split. On a class, numeric coercion yields the collection size; string coercion is deliberately rejected (a class is not a string, and silently stringifying one hides bugs). On an element, coercion resolves through the element's own value semantics.

Card.length;   // a number — the live instance count (numeric coercion works)
  +Card;         // the count, via @@toPrimitive('number')
  '' + Card;     // THROWS — string coercion of a class is intentionally disallowed

Symbol.toStringTag — the [object Tag] name

Implemented. Object.prototype.toString reflects the element's class name, and a class stringifies its tag with a Class suffix.

Object.prototype.toString.call(card);  // "[object Card]"

Symbol.match / matchAll / replace / search / split — the string protocol

Implemented (intentionally divergent). The five regex well-knowns are reinterpreted as collection operations: search = predicate, match = filter, matchAll = walker, replace = mutator, split = partitioner. This is a deliberate divergence from RegExp semantics — the receiver is the pattern, the query is the haystack, and a class answers with its living instances. All five verbs are pure except replace, which mutates matched text in the live DOM.

'apple'.search(Card);   // registry index of the first matching instance
  'apple'.match(Card);    // a Collection of matching instances
  'old'.replace(Card, 'new'); // mutates matching instances' text; returns the Collection

Symbol.isConcatSpreadable — flatten in concat

Implemented. Elements report as concat-spreadable, so array operations flatten Instance collections the way you'd expect.

Symbol.species — derived-object constructor

Implemented (pinned). Instance pins @@species non-configurable; element classes keep their inherited species so derived objects construct correctly. You rarely touch this directly — it's the plumbing that keeps subclassing well-behaved.

Symbol.dispose / Symbol.asyncDispose — using / await using

Stub (known). The slots exist so the explicit-resource-management protocol (using el = … / await using el = …) can light up where the platform supports it, but they are not currently wired to teardown. For disposal today, use the teardown protocol (el.dispose() / Instance.teardown()) or the Symbol.for('Instance.teardown') brand below.

// intended (not yet active):
  // using card = new Card();   // would auto-dispose at scope exit

  // today:
  card.dispose();               // explicit teardown — see the Teardown chapter

Reactivity

this.$field — reactive state

Description: Any property prefixed $ is a reactive field. Reads and writes route through the metaclass into a signal store held beside the element (weakly keyed on it) — the DOM node itself stays clean. Fields work on any minted element, with or without a class body, with or without compilation:

const d = new Div().appendTo(document.body);
  d.$count = 0;          // creates the reactive field
  d.$count;              // 0 — a tracked read inside any effect
  d.$count += 1;         // notifies subscribers
TierPrefixSemantics
shallow$The value is one signal. Object values notify per-child: subscribers track the properties they actually read.
deep$$The whole subtree is reactive — nested mutations (this.$$tree.a.b = 1) notify through the structure.

Writes are epoch-deduped: assigning the value a field already holds does not re-fire its subscribers. A changed value fires exactly once.

name$( ) { … } — effects (compiled classes)

Description: In compiled classes (.is sources, data-is, autoclass groups), a method whose name ends in $ is an effect: it runs once on mount to arm, tracking every $field it reads, and re-runs when its dependencies change.

class Counter extends Div {
    $n = 0
    show$() { this.textContent = 'n=' + this.$n }   // armed at mount; re-runs on writes
  }
  const c = new Counter().appendTo(document.body);
  c.textContent;   // "n=0"
  c.$n = 5;
  c.textContent;   // "n=5"

Two effects may read one field (both re-run per write); one effect may read two fields — but read the next entry before you rely on that.

Plain-JS classes: the $-suffix scan is a compiler feature. In un-compiled classes, arm effects at runtime with .effect(fn) — identical tracking, explicit arming.

Consensus & Δ — multi-dependency doctrine

Description: When one effect reads several plain $fields, it does not re-run on any write — it re-runs when every dependency has moved. This is the consensus barrier, and it is a feature: effects over multiple sources see a consistent frame, never a half-updated one.

class Pair extends Div {
    $a = 1
    $b = 2
    sum$() { this.textContent = String(this.$a + this.$b) }
  }
  const p = new Pair().appendTo(document.body);   // "3"
  p.$a = 10;      // barrier holds — still "3"
  p.$b = 20;      // both moved — ONE re-run, sees both: "30"

When you want any-source-fires (union) semantics, that is what the Δ combinator exists for — union is requested, never accidental. Derived computeds over several sources honor the same barrier through the chain.

The Chimera kernel

Chimera V17 is the signal engine everything above rides — a trit-kernel over a preallocated typed-array arena. Nodes (states, computeds, effects) are pointers into parallel lanes; dependency edges live in flat arrays; propagation is a walk over integers. That is why per-signal cost stays flat at scale and why full teardown is "drop the slab" — the whole graph is one allocation.

The kernel is discrete-first: its primitives are a valueless notify and a valueless subscribe; "current value + equality-dedup" is the store layer on top. That factoring is what lets one kernel carry both faces — behaviors ($fields: value at rest, dedup, replay) and pulses (DOM events: none of those) — each with the semantics its face requires.

The sigil grammar — position × family × depth

Description: One compact grammar generates the whole reactive vocabulary. Verbatim from the kernel's own charter: "Prefix=computed, suffix=effect; Δ=union(OR), $=consensus(AND); depth: @=atomic (reference-update), $=shallow, $$=deep; get X()=volatile, this.X(fn)=active."

Autonomous Parameterized Memoization

Deep reactivity ($$) is backed by APM — a deep-proxy cache that makes nested tracking both correct and cheap. Reading through a deep signal returns a proxy that tracks the exact path you touched; the proxies are memoized per object (a weak cache, with FIFO eviction under a size cap) so re-reads don't re-wrap, and a FinalizationRegistry disposes derived handles when their targets are collected.

el.$$tree = { user: { name: 'Ada', roles: ['admin'] } };

  el.effect(() => {
    render(el.$$tree.user.name);   // tracks ONLY …user.name —
  });                              // a write to …roles won't re-run this
  el.$$tree.user.name = 'Lin';     // re-runs; the path matched

"Autonomous" because you don't manage it — no memo keys, no invalidation calls. "Parameterized" because the cache is keyed on the object identities you actually traverse. The effect is that deep object state behaves like fine-grained signals without you declaring a signal per field.

Union vs consensus, deep. A deep proxy can track in union mode (any nested change fires) or consensus mode (the barrier holds until every touched path moved), matching the Δ/$ families at depth.

DOM Events

Events are the pulse face of signals: same kernel, discrete semantics. Every DOM event type is a first-class global class — Click, MouseDown, KeyUp, FocusIn, Wheel, PopState … — that constructs real native events and subscribes through one shared listener per target. 0.87.z

The vocabulary & naming

Names are Pascal renderings of the platform types, with compound boundaries resolved: dblclick → DblClick, contextmenu → ContextMenu, popstate → PopState, beforeunload → BeforeUnload, visibilitychange → VisibilityChange. Prefix families compose — MouseDown, PointerCancel, TouchStart, AnimationIteration — and only families that grant a bare form have one: Focus and Drag exist; Key, Mouse, Pointer deliberately do not.

Collision rule. If a name is already taken — by an element class or any global — the event class takes an Event postfix: input → InputEvent, select → SelectEvent, reset → ResetEvent, and storage → StorageEvent (the native Storage API trips it automatically). Every event class is always reachable at Instance.Events.*, whatever the globals decided.

Deriving events — handles: config · map · combine · sleep · chain 0.87.z

Description: 0.87's crown jewel. The bare call returns more than an inert event — it returns a handle: still a real native event (dispatchable, instanceof-true), now also a subscribable, derivable event signal. Teardown

Disposal is a first-class protocol, not a convention — symbol-keyed, one body serving every face, from a single element up to the whole framework. 0.87.v

el[ Instance.TEARDOWN ]( ) ⇒ Element · el[ Instance.TEARDOWN_ASYNC ]( ) ⇒ Promise

Description: The element face. The sync form disposes now; the async form awaits the element's @removal/@unmount transitions first. Both are exactly remove(true) / disposeAsync() under protocol names — the point is that the same symbols work at every level.

const el = new Card().appendTo(document.body);
  el[Instance.TEARDOWN]();          // disposed

  const t = new Toast().appendTo(document.body);
  await t[Instance.TEARDOWN_ASYNC]();   // exit transition honored, then disposed

Class[ Instance.TEARDOWN ]( ) · Class.teardown( ) · Class.teardownAsync( )

Description: The class face: self-destruct. Disposes every living instance of the class — and only that class; siblings and ancestors are untouched. Inherited by every subclass.

class A extends Div {}  class B extends A {}

  B.teardown();        // every B gone
  A.all().length;      // 1 — the A survived
  await A.teardownAsync();   // transitions honored per element

Description: The protocol aliases onto the platform's explicit-resource-management well-knowns, so using blocks dispose Instance elements automatically wherever the runtime ships them.

    // … work with temp …
  }   // scope exit → temp[Symbol.dispose]() → disposed
Realm note. The aliases bind at load to whatever the realm provides. Browsers with explicit resource management get the true well-knowns; environments without them (some embedded/VM realms) get inert shim keys — the protocol symbols above work everywhere regardless.

Instance.teardown( [unmint] ) ⇒ true

Description: The framework face. Disposes every managed element in the document, tears down the signal arena (Signal.teardown() — the slab is released; the next signal use re-bootstraps cleanly), and — with unmint = true — deletes every global the framework minted: the tag vocabulary, the event vocabulary, synthesized interfaces, Instance itself. The page returns to what it was.

  Instance.teardown(true);    // the nuclear option:
  typeof Div;                 // "undefined"
  typeof MouseDown;           // "undefined"
  typeof Instance;            // "undefined"
Own the page. Full teardown assumes every managed element is yours to destroy. One deliberate survivor: the IVC broker slot is non-configurable by design (tamper-proofing cuts both ways) — it remains, answering with the torn-down class object.

Interfaces

Three callable proxy interfaces live on every element — .worker, .socket, .listen — each a memoized, element-scoped machine: built on first touch, cached per element, and drained at disposal (workers terminate, sockets close, listeners unbind — with their element). Each interface is callable: the call is its primary verb; its named methods hang off the same handle. 0.87.v

FAQ — deeper cuts

Why a classic script and not a module?

Because the boot contract is the tag: attributes on document.currentScript configure the framework, and modules don't have a currentScript. It also makes the mental model honest — one file executes, the vocabulary exists. Module code interops through InstanceReady and the IVC broker; the Vite plugin packages both.

Where's the virtual DOM?

There isn't one, on purpose. Instance's bet is that the platform's real nodes plus a signal kernel beat a diffing layer: state changes run your effects, which touch exactly the properties you wrote. The benchmark numbers in Tooling are that bet, measured.

Can it coexist with React / another framework on the page?

In default mode, yes — Instance elements are real elements, foreign code sees nothing odd, and Instance claims globals only where names are free. Engine mode is the exception: it assumes page ownership. The jQuery merge exists because that coexistence is a feature, not an accident.

Server-side rendering?

Not a supported target at this version. Instance boots against a live document; headless DOMs (jsdom) run it fine for testing, but there is no hydration story — honesty over checkbox.

TypeScript?

No types ship at this version. The surface is deliberately platform-shaped — elements are their DOM types — so ambient typings are tractable, and a .its(Instance TypeScript) is on the roadmap rather than in the box.
